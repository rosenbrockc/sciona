# Phase 6: Hybrid Search -- Full-Text Search & Vector Embeddings

## Overview

Phase 6 adds hybrid search to the atom catalog: full-text search (FTS) via
PostgreSQL `tsvector` / GIN indexes for exact keyword and symbol lookup, and
vector similarity search via pgvector embeddings for semantic and
cross-discipline discovery. A hybrid ranking function combines both signals so
agent queries can exploit exact matches and fuzzy semantic relevance in a single
call.

The search infrastructure is layered on top of the catalog index materialized
view defined in Migration Plan Section 2.8.1. Phase 6 departs from the original
plan in one significant way: **embeddings are stored in a dedicated
`atom_embeddings` table rather than as a nullable column inside the materialized
view**. This avoids tying the embedding lifecycle to expensive matview refreshes
and allows independent model upgrades, partial re-embedding, and event-driven
refresh.

When Phase 6 is complete:

- Every publishable atom has a weighted `tsvector` search document indexed with
  GIN.
- Every publishable atom has a 1536-dimension embedding generated by
  `text-embedding-3-small`, stored in a dedicated table with an HNSW index.
- A single `search_atoms_hybrid` RPC supports `mode=fts|vector|hybrid` and is
  callable from PostgREST (`supabase.rpc()`).
- An `embedding_refresh_queue` table drives event-based re-embedding when atom
  content changes, replacing periodic batch sweeps.
- Python scripts generate and backfill embeddings using the OpenAI API.

## Step 2: Full-Text Search Setup

Reference: Migration Plan Section 2.8.1

The `catalog_atoms_index` materialized view already includes a `search_document`
column of type `tsvector`. This step ensures the column is correctly populated
and indexed.

### 2a. tsvector Column in the Materialized View

The matview definition from Phase 3 includes:

```sql
-- Inside catalog_atoms_index definition:
setweight(to_tsvector('english', a.fqdn), 'A') ||
setweight(to_tsvector('english', COALESCE(a.description, '')), 'B') ||
setweight(to_tsvector('english', COALESCE(d.content, '')), 'C') ||
setweight(to_tsvector('english', array_to_string(a.domain_tags, ' ')), 'B')
    AS search_document
```

Weight assignments:

| Weight | Source | Rationale |
|---|---|---|
| **A** (highest) | `fqdn` | Exact name matches should rank first. Agents frequently search by fully qualified name. |
| **B** | `atoms.description`, `domain_tags` | Technical description and domain tags are high-signal for keyword search. |
| **C** | Dejargonized description | Broader plain-language content is useful but should not outrank precise technical terms. |

### 2b. GIN Index on search_document

```sql
-- Already in Phase 3 matview DDL; confirmed here for completeness.
CREATE INDEX IF NOT EXISTS idx_catalog_atoms_index_fts
    ON public.catalog_atoms_index USING gin (search_document);
```

### 2c. FTS Search Function

```sql
-- ============================================================
-- FTS-only search: returns atoms ranked by ts_rank_cd
-- ============================================================

CREATE OR REPLACE FUNCTION public.search_atoms_fts(
    query_text TEXT,
    result_limit INTEGER DEFAULT 20,
    result_offset INTEGER DEFAULT 0
)
RETURNS TABLE (
    atom_id        UUID,
    fqdn           TEXT,
    technical_description TEXT,
    dejargonized_description TEXT,
    domain_tags    TEXT[],
    overall_verdict TEXT,
    risk_tier      TEXT,
    trust_readiness TEXT,
    fts_rank       REAL
)
LANGUAGE sql
STABLE
SECURITY INVOKER
SET search_path = ''
AS $$
    SELECT
        ci.atom_id,
        ci.fqdn,
        ci.technical_description,
        ci.dejargonized_description,
        ci.domain_tags,
        ci.overall_verdict,
        ci.risk_tier,
        ci.trust_readiness,
        ts_rank_cd(ci.search_document, websearch_to_tsquery('english', query_text)) AS fts_rank
    FROM public.catalog_atoms_index ci
    WHERE ci.search_document @@ websearch_to_tsquery('english', query_text)
    ORDER BY fts_rank DESC
    LIMIT result_limit
    OFFSET result_offset;
$$;
```

`SECURITY INVOKER` ensures RLS on the underlying `catalog_atoms_index` matview
is respected. Since the matview is pre-filtered to `status = 'approved' AND
is_publishable = TRUE`, additional per-row RLS checks are not needed for the
matview itself, but the invoker context ensures future policy changes propagate.

`websearch_to_tsquery` is used instead of `plainto_tsquery` because it supports
quoted phrases and boolean operators (`OR`, `-` for exclusion), which is useful
for agent queries like `"kalman filter" -extended`.

## Step 3: Atom Embeddings Table

Rather than storing the embedding vector inside the `catalog_atoms_index`
materialized view, we use a dedicated table. This gives several advantages:

1. **Independent refresh**: Embeddings can be updated per-atom without
   refreshing the entire matview.
2. **Model versioning**: The `model` column tracks which embedding model
   produced each vector, enabling incremental migration to a new model.
3. **Partial re-embedding**: Only atoms whose content changed need new
   embeddings, tracked via the refresh queue (Step 4).
4. **No matview schema lock-in**: Adding/removing embedding dimensions does not
   require dropping and recreating the matview.

### 3a. Table Definition

```sql
-- ============================================================
-- ATOM EMBEDDINGS (separate from matview for flexibility)
-- ============================================================

CREATE TABLE public.atom_embeddings (
    atom_id    UUID PRIMARY KEY REFERENCES public.atoms(atom_id) ON DELETE CASCADE,
    embedding  vector(1536) NOT NULL,
    model      TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    dimensions INTEGER NOT NULL DEFAULT 1536,
    -- The text that was embedded, stored for debugging and drift detection.
    -- Not indexed; only read during re-embedding comparisons.
    input_text_hash TEXT NOT NULL DEFAULT '',
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE public.atom_embeddings ENABLE ROW LEVEL SECURITY;

-- Embeddings are readable by anyone who can see the catalog index.
-- Write access is service-role only (embedding generation scripts).
CREATE POLICY atom_embeddings_select ON public.atom_embeddings
    FOR SELECT
    USING (
        EXISTS (
            SELECT 1 FROM public.atoms a
            WHERE a.atom_id = atom_embeddings.atom_id
              AND a.status = 'approved'
              AND a.is_publishable = TRUE
        )
    );
```

### 3b. Embedding Model Recommendation

| Property | Value |
|---|---|
| Model | `text-embedding-3-small` |
| Dimensions | 1536 |
| Provider | OpenAI |
| Cost | $0.020 / 1M tokens |
| Rationale | Best cost/quality tradeoff for catalog-scale (~500-2000 atoms). 1536 dimensions match the model's native output, avoiding the quality loss of dimension reduction. For a catalog of 505 atoms, the full embedding table is ~3 MB -- negligible. |

If the catalog grows beyond ~50k atoms or latency requirements tighten, consider
switching to `text-embedding-3-large` with `dimensions=1024` (Matryoshka
truncation) or a self-hosted model. The `model` column on `atom_embeddings`
supports this migration path.

### 3c. HNSW Index for Vector Similarity

```sql
-- HNSW index for approximate nearest neighbor search.
-- cosine distance is appropriate for normalized text embeddings.
CREATE INDEX idx_atom_embeddings_hnsw
    ON public.atom_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

HNSW parameter choices:

| Parameter | Value | Rationale |
|---|---|---|
| `m` | 16 | Default; good recall/speed balance for catalogs under 100k rows. |
| `ef_construction` | 64 | Default; higher values improve recall at build time but the catalog is small enough that this is fast regardless. |
| Distance operator | `vector_cosine_ops` | Cosine similarity is standard for normalized text embeddings from OpenAI. |

At query time, `SET LOCAL hnsw.ef_search = 40` (the default) provides >95%
recall for catalogs under 10k rows. Increase to 100-200 if recall drops as the
catalog grows.

### 3d. Vector Search Function

```sql
-- ============================================================
-- Vector-only search: returns atoms ranked by cosine similarity
-- ============================================================

CREATE OR REPLACE FUNCTION public.search_atoms_vector(
    query_embedding vector(1536),
    result_limit INTEGER DEFAULT 20,
    similarity_threshold DOUBLE PRECISION DEFAULT 0.3
)
RETURNS TABLE (
    atom_id        UUID,
    fqdn           TEXT,
    technical_description TEXT,
    dejargonized_description TEXT,
    domain_tags    TEXT[],
    overall_verdict TEXT,
    risk_tier      TEXT,
    trust_readiness TEXT,
    similarity     DOUBLE PRECISION
)
LANGUAGE sql
STABLE
SECURITY INVOKER
SET search_path = ''
AS $$
    SELECT
        ci.atom_id,
        ci.fqdn,
        ci.technical_description,
        ci.dejargonized_description,
        ci.domain_tags,
        ci.overall_verdict,
        ci.risk_tier,
        ci.trust_readiness,
        1 - (ae.embedding <=> query_embedding) AS similarity
    FROM public.atom_embeddings ae
    JOIN public.catalog_atoms_index ci ON ci.atom_id = ae.atom_id
    WHERE 1 - (ae.embedding <=> query_embedding) >= similarity_threshold
    ORDER BY ae.embedding <=> query_embedding
    LIMIT result_limit;
$$;
```

The caller is responsible for generating the query embedding client-side (or via
a thin server-side wrapper) before calling this function. This keeps the database
free of external API dependencies.

## Step 4: Embedding Refresh Queue

Instead of a periodic cron job that re-embeds all atoms on a schedule, we use an
event-driven queue. When atom content changes (description, dejargonized text,
domain tags, fqdn), a trigger enqueues the atom for re-embedding. A worker
process drains the queue.

### 4a. Queue Table

```sql
-- ============================================================
-- EMBEDDING REFRESH QUEUE
-- Event-driven queue for atoms that need (re-)embedding.
-- ============================================================

CREATE TABLE public.embedding_refresh_queue (
    queue_id    BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    atom_id     UUID NOT NULL REFERENCES public.atoms(atom_id) ON DELETE CASCADE,
    reason      TEXT NOT NULL DEFAULT 'content_changed',
    status      TEXT NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
    error_message TEXT NOT NULL DEFAULT '',
    enqueued_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    started_at  TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    attempts    INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_embedding_queue_pending
    ON public.embedding_refresh_queue (status, enqueued_at)
    WHERE status = 'pending';

CREATE INDEX idx_embedding_queue_atom
    ON public.embedding_refresh_queue (atom_id);

-- RLS: service role only. No end-user access.
ALTER TABLE public.embedding_refresh_queue ENABLE ROW LEVEL SECURITY;
```

### 4b. Enqueue Trigger

Fires when content that affects the embedding changes on the `atoms` table or
the `atom_descriptions` table.

```sql
CREATE OR REPLACE FUNCTION public.enqueue_embedding_refresh()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER SET search_path = ''
AS $$
DECLARE
    target_atom_id UUID;
BEGIN
    target_atom_id := COALESCE(NEW.atom_id, OLD.atom_id);

    -- Avoid duplicate pending entries for the same atom.
    IF NOT EXISTS (
        SELECT 1
        FROM public.embedding_refresh_queue
        WHERE atom_id = target_atom_id
          AND status = 'pending'
    ) THEN
        INSERT INTO public.embedding_refresh_queue (atom_id, reason)
        VALUES (target_atom_id, TG_ARGV[0]);
    END IF;

    RETURN NULL;
END;
$$;

-- Trigger on atoms: fires when fqdn, description, or domain_tags change.
CREATE TRIGGER trg_embedding_refresh_atoms
    AFTER INSERT OR UPDATE OF fqdn, description, domain_tags ON public.atoms
    FOR EACH ROW
    EXECUTE FUNCTION public.enqueue_embedding_refresh('atom_content_changed');

-- Trigger on atom_descriptions: fires when dejargonized content changes.
CREATE TRIGGER trg_embedding_refresh_descriptions
    AFTER INSERT OR UPDATE OF content ON public.atom_descriptions
    FOR EACH ROW
    EXECUTE FUNCTION public.enqueue_embedding_refresh('description_changed');
```

### 4c. Queue Cleanup

Old completed/failed entries should be purged periodically. A simple SQL
statement run by `pg_cron` or the embedding worker:

```sql
-- Purge completed queue entries older than 30 days.
DELETE FROM public.embedding_refresh_queue
WHERE status IN ('completed', 'failed')
  AND completed_at < now() - INTERVAL '30 days';
```

## Step 5: Hybrid Search Function

The hybrid search function combines FTS and vector similarity scores using
reciprocal rank fusion (RRF). RRF is preferred over a simple weighted sum
because it is robust to score-scale differences between FTS ranks and cosine
similarities.

### 5a. Hybrid Search RPC

```sql
-- ============================================================
-- HYBRID SEARCH: combines FTS + vector similarity via RRF
-- Callable from PostgREST: supabase.rpc('search_atoms_hybrid', {...})
-- ============================================================

CREATE OR REPLACE FUNCTION public.search_atoms_hybrid(
    query_text TEXT,
    query_embedding vector(1536) DEFAULT NULL,
    mode TEXT DEFAULT 'hybrid',
    result_limit INTEGER DEFAULT 20,
    result_offset INTEGER DEFAULT 0,
    fts_weight DOUBLE PRECISION DEFAULT 1.0,
    vector_weight DOUBLE PRECISION DEFAULT 1.0,
    rrf_k INTEGER DEFAULT 60,
    similarity_threshold DOUBLE PRECISION DEFAULT 0.3
)
RETURNS TABLE (
    atom_id        UUID,
    fqdn           TEXT,
    technical_description TEXT,
    dejargonized_description TEXT,
    domain_tags    TEXT[],
    overall_verdict TEXT,
    risk_tier      TEXT,
    trust_readiness TEXT,
    fts_rank       REAL,
    similarity     DOUBLE PRECISION,
    hybrid_score   DOUBLE PRECISION
)
LANGUAGE plpgsql
STABLE
SECURITY INVOKER
SET search_path = ''
AS $$
BEGIN
    -- FTS-only mode
    IF mode = 'fts' THEN
        RETURN QUERY
        SELECT
            ci.atom_id,
            ci.fqdn,
            ci.technical_description,
            ci.dejargonized_description,
            ci.domain_tags,
            ci.overall_verdict,
            ci.risk_tier,
            ci.trust_readiness,
            ts_rank_cd(ci.search_document, websearch_to_tsquery('english', query_text)) AS fts_rank,
            NULL::DOUBLE PRECISION AS similarity,
            ts_rank_cd(ci.search_document, websearch_to_tsquery('english', query_text))::DOUBLE PRECISION AS hybrid_score
        FROM public.catalog_atoms_index ci
        WHERE ci.search_document @@ websearch_to_tsquery('english', query_text)
        ORDER BY fts_rank DESC
        LIMIT result_limit
        OFFSET result_offset;
        RETURN;
    END IF;

    -- Vector-only mode
    IF mode = 'vector' THEN
        IF query_embedding IS NULL THEN
            RAISE EXCEPTION 'query_embedding is required for vector mode';
        END IF;
        RETURN QUERY
        SELECT
            ci.atom_id,
            ci.fqdn,
            ci.technical_description,
            ci.dejargonized_description,
            ci.domain_tags,
            ci.overall_verdict,
            ci.risk_tier,
            ci.trust_readiness,
            NULL::REAL AS fts_rank,
            1 - (ae.embedding <=> query_embedding) AS similarity,
            (1 - (ae.embedding <=> query_embedding)) AS hybrid_score
        FROM public.atom_embeddings ae
        JOIN public.catalog_atoms_index ci ON ci.atom_id = ae.atom_id
        WHERE 1 - (ae.embedding <=> query_embedding) >= similarity_threshold
        ORDER BY ae.embedding <=> query_embedding
        LIMIT result_limit
        OFFSET result_offset;
        RETURN;
    END IF;

    -- Hybrid mode (default): reciprocal rank fusion
    IF query_embedding IS NULL THEN
        RAISE EXCEPTION 'query_embedding is required for hybrid mode';
    END IF;

    RETURN QUERY
    WITH fts_results AS (
        SELECT
            ci.atom_id,
            ROW_NUMBER() OVER (ORDER BY ts_rank_cd(ci.search_document, websearch_to_tsquery('english', query_text)) DESC) AS fts_row_rank,
            ts_rank_cd(ci.search_document, websearch_to_tsquery('english', query_text)) AS fts_rank_score
        FROM public.catalog_atoms_index ci
        WHERE ci.search_document @@ websearch_to_tsquery('english', query_text)
    ),
    vector_results AS (
        SELECT
            ae.atom_id,
            ROW_NUMBER() OVER (ORDER BY ae.embedding <=> query_embedding) AS vec_row_rank,
            1 - (ae.embedding <=> query_embedding) AS vec_similarity
        FROM public.atom_embeddings ae
        JOIN public.catalog_atoms_index ci ON ci.atom_id = ae.atom_id
        WHERE 1 - (ae.embedding <=> query_embedding) >= similarity_threshold
    ),
    combined AS (
        SELECT
            COALESCE(f.atom_id, v.atom_id) AS atom_id,
            f.fts_rank_score,
            v.vec_similarity,
            -- RRF: score = sum of 1/(k + rank) across both retrieval methods
            COALESCE(fts_weight * (1.0 / (rrf_k + f.fts_row_rank)), 0) +
            COALESCE(vector_weight * (1.0 / (rrf_k + v.vec_row_rank)), 0) AS rrf_score
        FROM fts_results f
        FULL OUTER JOIN vector_results v ON f.atom_id = v.atom_id
    )
    SELECT
        ci.atom_id,
        ci.fqdn,
        ci.technical_description,
        ci.dejargonized_description,
        ci.domain_tags,
        ci.overall_verdict,
        ci.risk_tier,
        ci.trust_readiness,
        c.fts_rank_score::REAL AS fts_rank,
        c.vec_similarity AS similarity,
        c.rrf_score AS hybrid_score
    FROM combined c
    JOIN public.catalog_atoms_index ci ON ci.atom_id = c.atom_id
    ORDER BY c.rrf_score DESC
    LIMIT result_limit
    OFFSET result_offset;
END;
$$;
```

### 5b. PostgREST Usage

From the Supabase Python client:

```python
# FTS-only search (no embedding needed)
result = supabase.rpc("search_atoms_hybrid", {
    "query_text": "kalman filter",
    "mode": "fts",
    "result_limit": 10,
}).execute()

# Vector-only search (requires pre-computed query embedding)
query_emb = openai_client.embeddings.create(
    model="text-embedding-3-small",
    input="time series anomaly detection for sensor data",
).data[0].embedding

result = supabase.rpc("search_atoms_hybrid", {
    "query_text": "",
    "query_embedding": query_emb,
    "mode": "vector",
    "result_limit": 10,
}).execute()

# Hybrid search (both FTS and vector)
result = supabase.rpc("search_atoms_hybrid", {
    "query_text": "kalman filter",
    "query_embedding": query_emb,
    "mode": "hybrid",
    "result_limit": 20,
    "fts_weight": 1.0,
    "vector_weight": 0.8,
}).execute()
```

The `fts_weight` and `vector_weight` parameters allow callers to bias results.
For agent queries where the search term is a natural-language description,
increase `vector_weight`. For exact symbol/name lookups, increase `fts_weight`
or use `mode=fts`.

## Step 6: Python Embedding Scripts

### 6a. Embedding Generator

Generates embeddings for atoms and upserts them into `atom_embeddings`. Used for
both initial backfill and queue-driven refresh.

```python
# scripts/generate_embeddings.py
"""Generate and upsert atom embeddings using OpenAI text-embedding-3-small.

Usage:
    # Backfill all atoms that lack embeddings:
    python scripts/generate_embeddings.py --backfill

    # Process pending items from the refresh queue:
    python scripts/generate_embeddings.py --drain-queue

    # Re-embed a specific atom:
    python scripts/generate_embeddings.py --atom-id <uuid>

Requires:
    OPENAI_API_KEY          - OpenAI API key
    SUPABASE_URL            - Supabase project URL
    SUPABASE_SERVICE_KEY    - Supabase service role key
"""

from __future__ import annotations

import argparse
import hashlib
import logging
import os
import time
from typing import Any

from openai import OpenAI
from supabase import create_client

logger = logging.getLogger(__name__)

EMBEDDING_MODEL = "text-embedding-3-small"
EMBEDDING_DIMENSIONS = 1536
BATCH_SIZE = 100  # OpenAI supports up to 2048 inputs per request


def build_embedding_input(atom: dict[str, Any]) -> str:
    """Assemble the text that will be embedded for a single atom.

    Mirrors the tsvector construction in catalog_atoms_index but as plain text
    so the embedding captures the same semantic content.
    """
    parts = [
        atom.get("fqdn", ""),
        atom.get("technical_description", "") or "",
        atom.get("dejargonized_description", "") or "",
        " ".join(atom.get("domain_tags", []) or []),
    ]
    return "\n".join(p for p in parts if p)


def compute_input_hash(text: str) -> str:
    return hashlib.sha256(text.encode()).hexdigest()[:16]


def embed_batch(client: OpenAI, texts: list[str]) -> list[list[float]]:
    """Call OpenAI embeddings API for a batch of texts."""
    response = client.embeddings.create(
        model=EMBEDDING_MODEL,
        input=texts,
        dimensions=EMBEDDING_DIMENSIONS,
    )
    return [item.embedding for item in response.data]


def backfill(supabase: Any, openai_client: OpenAI) -> None:
    """Generate embeddings for all publishable atoms that lack one."""
    atoms = (
        supabase.rpc("get_atoms_needing_embeddings").execute().data
    )
    if not atoms:
        logger.info("No atoms need embeddings.")
        return

    logger.info("Generating embeddings for %d atoms", len(atoms))
    for i in range(0, len(atoms), BATCH_SIZE):
        batch = atoms[i : i + BATCH_SIZE]
        texts = [build_embedding_input(a) for a in batch]
        hashes = [compute_input_hash(t) for t in texts]
        embeddings = embed_batch(openai_client, texts)

        rows = [
            {
                "atom_id": a["atom_id"],
                "embedding": emb,
                "model": EMBEDDING_MODEL,
                "dimensions": EMBEDDING_DIMENSIONS,
                "input_text_hash": h,
            }
            for a, emb, h in zip(batch, embeddings, hashes)
        ]
        supabase.table("atom_embeddings").upsert(rows).execute()
        logger.info(
            "Upserted %d embeddings (batch %d)", len(rows), i // BATCH_SIZE + 1
        )

        # Respect rate limits
        time.sleep(0.5)


def drain_queue(supabase: Any, openai_client: OpenAI) -> None:
    """Process pending entries from embedding_refresh_queue."""
    pending = (
        supabase.table("embedding_refresh_queue")
        .select("queue_id, atom_id")
        .eq("status", "pending")
        .order("enqueued_at")
        .limit(BATCH_SIZE)
        .execute()
        .data
    )
    if not pending:
        logger.info("No pending queue entries.")
        return

    queue_ids = [p["queue_id"] for p in pending]
    atom_ids = [p["atom_id"] for p in pending]

    # Mark as processing
    supabase.table("embedding_refresh_queue").update(
        {"status": "processing", "started_at": "now()"}
    ).in_("queue_id", queue_ids).execute()

    # Fetch atom content for embedding
    atoms = (
        supabase.table("catalog_atoms_index")
        .select(
            "atom_id, fqdn, technical_description, "
            "dejargonized_description, domain_tags"
        )
        .in_("atom_id", atom_ids)
        .execute()
        .data
    )
    atom_map = {a["atom_id"]: a for a in atoms}

    succeeded = []
    failed = []
    for qid, aid in zip(queue_ids, atom_ids):
        atom = atom_map.get(aid)
        if not atom:
            failed.append((qid, "atom not found in catalog index"))
            continue

        text = build_embedding_input(atom)
        input_hash = compute_input_hash(text)

        try:
            emb = embed_batch(openai_client, [text])[0]
            supabase.table("atom_embeddings").upsert({
                "atom_id": aid,
                "embedding": emb,
                "model": EMBEDDING_MODEL,
                "dimensions": EMBEDDING_DIMENSIONS,
                "input_text_hash": input_hash,
            }).execute()
            succeeded.append(qid)
        except Exception as exc:
            failed.append((qid, str(exc)))

    if succeeded:
        supabase.table("embedding_refresh_queue").update(
            {"status": "completed", "completed_at": "now()"}
        ).in_("queue_id", succeeded).execute()
    for qid, err in failed:
        supabase.table("embedding_refresh_queue").update(
            {"status": "failed", "error_message": err, "completed_at": "now()"}
        ).eq("queue_id", qid).execute()

    logger.info(
        "Queue drain: %d succeeded, %d failed", len(succeeded), len(failed)
    )


def main() -> None:
    logging.basicConfig(level=logging.INFO)
    parser = argparse.ArgumentParser()
    group = parser.add_mutually_exclusive_group(required=True)
    group.add_argument("--backfill", action="store_true")
    group.add_argument("--drain-queue", action="store_true")
    group.add_argument("--atom-id", type=str)
    args = parser.parse_args()

    supabase = create_client(
        os.environ["SUPABASE_URL"],
        os.environ["SUPABASE_SERVICE_KEY"],
    )
    openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

    if args.backfill:
        backfill(supabase, openai_client)
    elif args.drain_queue:
        drain_queue(supabase, openai_client)
    elif args.atom_id:
        # Single-atom mode: fetch, embed, upsert
        atom = (
            supabase.table("catalog_atoms_index")
            .select(
                "atom_id, fqdn, technical_description, "
                "dejargonized_description, domain_tags"
            )
            .eq("atom_id", args.atom_id)
            .single()
            .execute()
            .data
        )
        if not atom:
            logger.error("Atom %s not found in catalog index", args.atom_id)
            return
        text = build_embedding_input(atom)
        emb = embed_batch(openai_client, [text])[0]
        supabase.table("atom_embeddings").upsert({
            "atom_id": args.atom_id,
            "embedding": emb,
            "model": EMBEDDING_MODEL,
            "dimensions": EMBEDDING_DIMENSIONS,
            "input_text_hash": compute_input_hash(text),
        }).execute()
        logger.info("Embedded atom %s", args.atom_id)


if __name__ == "__main__":
    main()
```

### 6b. Helper RPC: Atoms Needing Embeddings

Used by the backfill script to find publishable atoms without an embedding or
whose content has changed since the last embedding.

```sql
CREATE OR REPLACE FUNCTION public.get_atoms_needing_embeddings()
RETURNS TABLE (

    fqdn TEXT,
    technical_description TEXT,
    dejargonized_description TEXT,
    domain_tags TEXT[]
)
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = ''

    SELECT
        ci.atom_id,
        ci.fqdn,
        ci.technical_description,
        ci.dejargonized_description,
        ci.domain_tags
    FROM public.catalog_atoms_index ci
    LEFT JOIN public.atom_embeddings ae ON ae.atom_id = ci.atom_id
    WHERE ae.atom_id IS NULL
    ORDER BY ci.fqdn;
$$;
```

## Step 7: Matview Refresh Considerations

The `catalog_atoms_index` materialized view must be refreshed when underlying
data changes. This is orthogonal to embedding refresh:

- **Matview refresh**: Updates FTS search documents and catalog metadata.
  Triggered after bulk data changes (atom creation, audit rollup updates,
  description changes). Use `REFRESH MATERIALIZED VIEW CONCURRENTLY
  public.catalog_atoms_index;` to avoid locking reads.
- **Embedding refresh**: Updates vector embeddings. Driven by the
  `embedding_refresh_queue`. Happens independently of the matview refresh.

Recommended approach: refresh the matview after any batch operation that changes
atoms, descriptions, or audit rollups, then run `scripts/generate_embeddings.py
--drain-queue` to pick up any embedding changes.

```sql
-- Convenience function for application code to trigger matview refresh.
CREATE OR REPLACE FUNCTION public.refresh_catalog_index()
RETURNS VOID
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = ''
AS $$
BEGIN
    REFRESH MATERIALIZED VIEW CONCURRENTLY public.catalog_atoms_index;
END;
$$;
```
